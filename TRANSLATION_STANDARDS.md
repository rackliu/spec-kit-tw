# 翻譯品質標準

本文件定義了 Spec Kit TW 專案的翻譯品質標準和檢查清單, 確保中文版與原版保持完全一致的功能和使用者體驗.

## 翻譯原則

### 核心原則
1. **功能對等**: 翻譯後功能必須與原版完全一致
2. **使用者導向**: 面向中文開發者, 提供友好的母語介面
3. **技術保留**: 程式碼層面保持英文, 確保技術準確性
4. **語義一致**: 不能新增或減少內容, 保持原版語義

### 翻譯範圍

**需要完全中文字地化的內容**: 
- 使用者文件: README.md、spec-driven.md、docs/ 目錄
- 模板系統: templates/ 和 templates/commands/ 目錄下的所有檔案
- 文件: docs/ 目錄下的所有檔案
- 專案章程: memory/constitution.md(包括佔位符和說明文字)
- CLI 介面: src/specify_cli/ 中的輸出資訊、幫助文字、錯誤訊息
- 專案級檔案: CONTRIBUTING.md、SUPPORT.md、SECURITY.md、CODE_OF_CONDUCT.md

**保持英文不翻譯的內容**:
- 構建指令碼: scripts/ 目錄(完全同步原版)
- 開發環境: .devcontainer/ 目錄(完全同步原版)
- 媒體資源: media/ 目錄(完全同步原版)
- CI配置: .github/ 目錄(謹慎同步, 不翻譯)
- 程式碼層面: 變數名、函式名、類名等識別符號
- 章程佔位符: 如 [PROJECT_NAME]、[PRINCIPLE_1_NAME] 等(保持原格式)
- **技術標記**: 指令碼依賴的關鍵標記，詳見"不可翻譯技術標記"章節

## 術語處理標準

- 嚴格按照術語表 [@術語表](TERMINOLOGY.md) 執行翻譯
- 對於新出現的術語, 詢問使用者後補充到術語表

### 中英文混排規則
```markdown
✅ 推薦: 
- 使用 Claude Code CLI 進行開發
- 支援 JSON 格式的設定檔
- 透過 GitHub API 獲取資料

❌ 避免: 
- 使用Claude Code CLI進行開發
- 支援 JSON 格式的設定檔
```

### 命令替換規則
- 原文中的 `specify` 命令 → `specify-tw`
- 包名 `specify-cli` → `specify-tw-cli`
- 路徑中的 `.specify` 開頭保持不變

### 路徑和斜槓命令處理準則

**重要說明**: 路徑處理由釋出指令碼 `create-release-packages.sh` 中的 `rewrite_paths` 函式統一處理，翻譯時不應修改原版路徑格式。

#### 釋出時的路徑轉換
釋出指令碼會自動將以下路徑轉換為 `.specify/` 字首格式：
- `memory/` → `.specify/memory/`
- `scripts/` → `.specify/scripts/`
- `templates/` → `.specify/templates/`

#### 翻譯時的處理原則
- **保持原樣**: 翻譯時保持原版路徑格式不變
- **不新增字首**: 不要在翻譯時手動新增 `.specify/` 字首
- **避免重複**: 防止釋出時出現 `.specify.specify/` 的重複字首問題

#### 正確範例
```markdown
✅ 翻譯時保持原樣:
- 讀取 `memory/constitution.md` 中的專案章程
- 參考 `templates/plan-template.md` 模板
- 執行 `scripts/setup.sh` 指令碼

❌ 翻譯時錯誤新增字首:
- 讀取 `.specify/memory/constitution.md` 中的專案章程
- 參考 `.specify/templates/plan-template.md` 模板
- 執行 `.specify/scripts/setup.sh` 指令碼
```

#### 斜槓命令處理
- **保持原版格式**: 斜槓命令格式與原版保持一致
- **不新增字首**: 不要在翻譯時新增 `/speckit.` 字首
- **釋出時處理**: 斜槓命令的字首由釋出指令碼統一處理

#### 驗證清單
- [ ] 翻譯檔案中的路徑保持原版格式，未新增 `.specify/` 字首
- [ ] 斜槓命令保持原版格式，未新增 `/speckit.` 字首
- [ ] 確保釋出時不會產生重複的 `.specify.specify/` 路徑
- [ ] 所有內部引用與原版保持一致

## 不可翻譯技術標記

### 重要說明
以下標記是**指令碼依賴的技術標記**，必須保持英文格式，否則會導致自動化工具失效。

### 核心技術標記列表

| 標記 | 用途 | 出現位置 | 影響指令碼 |
|------|------|----------|----------|
| `NEEDS CLARIFICATION` | 標記不明確的規劃項 | 模板檔案、計劃文件 | update-agent-context.sh/.ps1 |
| `N/A` | 標記不適用項 | 模板檔案、計劃文件 | update-agent-context.sh/.ps1 |
| `TODO` | 標記待辦事項 | 程式碼註釋、文件 | 各種構建指令碼 |
| `TKTK` | 標記佔位符 | 模板、文件 | 分析工具 |
| `???` | 標記不確定內容 | 模板、文件 | 分析工具 |

### 嚴格遵守規則

#### ✅ 正確處理
- **保持原樣**: `NEEDS CLARIFICATION`、`N/A` 等標記保持英文
- **解釋說明**: 在使用者介面文字中用中文解釋標記含義
- **一致使用**: 在所有模板和文件中使用相同格式

#### ❌ 嚴禁操作
- **禁止翻譯**: 不得翻譯為 `需要澄清`、`不適用` 等
- **禁止修改**: 不得更改標記格式或大小寫
- **停用變體**: 不得使用如 `NEED-CLARIFICATION` 等變體

### 驗證方法

1. **指令碼驗證**: 使用 `scripts/bash/update-agent-context.sh` 測試
2. **搜尋檢查**: 確保指令碼依賴的標記未被翻譯
3. **功能測試**: 驗證標記過濾邏輯正常工作
4. **自動化檢查**: 整合到翻譯 QA 流程中

## 錯誤分類

### 嚴重錯誤(必須修復)
1. **功能錯誤**: 翻譯導致功能異常
2. **資訊缺失**: 遺漏重要內容
3. **術語錯誤**: 技術術語翻譯錯誤
4. **路徑錯誤**: 路徑或連結錯誤

### 重要錯誤(建議修復)
1. **術語不一致**: 同一術語在不同檔案中翻譯不同
2. **格式問題**: 影響閱讀的格式錯誤
3. **語句不通**: 影響理解的語法或表達問題

### 一般錯誤(可選修復)
1. **標點符號**: 中英文標點混用問題
2. **語言風格**: 輕微的表達不夠最佳化
3. **格式細節**: 不影響功能的格式問題

## 輸出報告結構

翻譯審查報告應包含以下部分: 

1. **執行摘要**
   - 檢查的檔案總數
   - 發現的問題總數
   - 嚴重錯誤數量
   - 整體品質評估

2. **詳細問題列表**
   - 按檔案分類的問題
   - 問題描述和位置
   - 錯誤級別和修復建議

3. **術語一致性檢查**
   - 術語使用情況統計
   - 不一致術語列表

4. **功能驗證結果**
   - CLI 功能測試結果
   - 模板檔案驗證結果

5. **修復建議**
   - 優先修復的問題列表
   - 具體的修復方案

## 特殊檔案翻譯規則

### checklist.md 混合型文件翻譯標準

`templates/commands/checklist.md` 是混合型文件, 包含使用者指導和 AI 系統指令, 需分類處理: 

#### ✅ 必須翻譯的內容
- 使用者介面標題和說明文字
- 操作步驟和重要說明
- 品質維度名稱(完整性、清晰度、一致性等)
- 範例的描述和解釋文字

#### ❌ 保持英文的內容
- YAML Front matter 配置
- Generation algorithm(AI 演算法邏輯)
- 技術實現指令和具體步驟
- 範例專案內容和程式碼塊
- 系統標記(`[Completeness]`, `[Spec §X.Y]` 等)
- 變數佔位符(`$ARGUMENTS`, `{SCRIPT}`)

#### 🔧 特殊處理
- 混合段落: 翻譯說明文字, 保持範例英文
- 技術術語: `API`, `CLI`, `Git` 等保持英文
- 格式保護: 不破壞 Markdown 格式和系統標記
- 中文中始終使用你而不是您
